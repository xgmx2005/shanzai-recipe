# 膳哉知识库增强推荐 Agent 设计文档

## 1. 设计结论

膳哉当前版本不做“AI 随机生成完整新菜谱”。系统定位调整为：

> 基于真实菜谱知识库、健康档案和规则评分的 AI 菜谱推荐 Agent。

AI 的职责是解释和增强推荐结果，而不是凭空创造菜谱主数据。菜谱名称、图片、食材、营养值、步骤和购物清单计算都必须来自数据库知识库。

## 2. 为什么不直接生成随机菜谱

直接让 AI 生成菜谱会带来以下工程风险：

- 图片无法可靠对应：AI 随机生成的菜名和已有图片没有稳定映射。
- 营养数据不可控：AI 可能编造热量、蛋白质和用量。
- 购物清单不可信：AI 生成的食材可能不在食材库中，无法合并、分类和勾选。
- 历史记录不可复现：同一输入可能生成不同菜谱，演示和调试都不稳定。
- 管理端失去价值：如果菜谱都由 AI 临时生成，菜谱库和食材库维护模块会变弱。

所以当前版本采用“数据库是真实知识库，AI 是解释层”的方案。

## 3. 当前版本目标

用户输入已有食材、排除食材、健康目标、烹饪时间和人数后，系统按以下流程工作：

1. 从数据库读取上架菜谱和菜谱食材关系。
2. 根据用户健康档案合并忌口和过敏食材。
3. 对菜谱进行硬过滤，排除含禁忌食材的菜谱。
4. 用规则评分计算食材命中、健康目标、口味偏好和时间匹配。
5. 选出 3-5 道真实菜谱。
6. 将用户输入和推荐结果摘要交给 DeepSeek。
7. DeepSeek 只返回结构化推荐分析：
   - `summary`: 本次推荐整体说明。
   - `healthTip`: 健康建议。
   - `shoppingTip`: 购物清单建议。
   - `topRecipeReason`: 第一道推荐菜谱的个性化理由。
8. 前端展示 AI 分析和数据库菜谱卡片。
9. 用户点击菜谱详情时仍读取数据库菜谱详情。
10. 用户生成购物清单时仍基于 `recipe_ingredient` 和 `ingredient` 计算缺少项。

## 4. 产品边界

### 4.1 当前版本做

- AI 解释推荐结果。
- AI 说明为什么这些菜适合当前目标。
- AI 说明已有食材命中情况。
- AI 给出健康饮食建议。
- AI 给出购物清单使用建议。
- 保留 AI 不可用时的本地规则兜底。
- 前端显示本次是否为 AI 生成分析。

### 4.2 当前版本不做

- 不让 AI 直接创建正式菜谱。
- 不让 AI 生成菜谱图片。
- 不让 AI 编造营养数据。
- 不让 AI 编造购物清单项。
- 不把 AI 输出直接写入 `recipe` 表。
- 不做图片上传和图片审核流程。

### 4.3 后续可扩展

如果后期时间充足，可新增“AI 菜谱草稿”能力，但必须单独建模：

- AI 生成菜谱草稿，状态为 `DRAFT_AI`。
- 草稿不进入普通推荐结果。
- 草稿图片使用默认占位图或由维护员上传。
- 维护员审核后才能发布为正式菜谱。

该能力不属于当前版本。

## 5. 后端设计

### 5.1 推荐响应字段

`POST /api/recommendations` 的响应从当前结构：

```json
{
  "historyId": 17,
  "aiSummary": "今天推荐低脂高蛋白搭配，适合控制热量。",
  "recipes": []
}
```

扩展为：

```json
{
  "historyId": 17,
  "aiSummary": "你现有鸡胸肉和西兰花，系统优先推荐低脂高蛋白、烹饪时间较短的轻食组合。",
  "aiHealthTip": "这组推荐蛋白质较充足，减脂期建议控制额外油脂，并搭配足量蔬菜提升饱腹感。",
  "aiShoppingTip": "购物清单会自动排除你已有的鸡胸肉和西兰花，只补充玉米、鸡蛋等缺少食材。",
  "aiGenerated": true,
  "recipes": [
    {
      "id": 1,
      "name": "鸡胸肉西兰花轻食碗",
      "score": 92,
      "reason": "这道菜直接命中鸡胸肉和西兰花，蛋白质充足、脂肪较低，适合作为减脂期的一餐。",
      "calories": 420,
      "protein": 35.00,
      "imageUrl": "/images/recipes/chicken-broccoli-bowl.jpg"
    }
  ]
}
```

字段说明：

| 字段 | 来源 | 说明 |
|---|---|---|
| `historyId` | 数据库 | 推荐历史 ID |
| `aiSummary` | DeepSeek 或本地兜底 | 本次推荐整体解释 |
| `aiHealthTip` | DeepSeek 或本地兜底 | 健康建议 |
| `aiShoppingTip` | DeepSeek 或本地兜底 | 购物清单建议 |
| `aiGenerated` | 后端判断 | true 表示本次文案来自 DeepSeek，false 表示本地兜底 |
| `recipes` | 数据库 + 规则评分 | 推荐菜谱列表 |

### 5.2 数据库持久化

`recommendation_history` 增加字段：

```sql
ALTER TABLE recommendation_history
  ADD COLUMN ai_health_tip TEXT AFTER ai_summary,
  ADD COLUMN ai_shopping_tip TEXT AFTER ai_health_tip,
  ADD COLUMN ai_generated TINYINT NOT NULL DEFAULT 0 AFTER ai_shopping_tip;
```

`schema.sql` 同步更新，避免重建数据库时丢字段。

### 5.3 DeepSeek 输入上下文

后端传给 DeepSeek 的上下文必须来自可信数据：

- 用户已有食材。
- 用户排除食材。
- 用户饮食目标。
- 烹饪时间。
- 推荐出的菜谱名称、分数、热量、蛋白质、命中食材和健康标签。

DeepSeek 不接收完整数据库，不负责重新排序，只解释后端已经算出的推荐结果。

### 5.4 DeepSeek 输出约束

DeepSeek 必须返回 JSON，不允许 Markdown：

```json
{
  "summary": "本次推荐总结",
  "healthTip": "健康提示",
  "shoppingTip": "购物清单提示",
  "topRecipeReason": "第一道菜的个性化推荐理由"
}
```

后端解析失败、网络超时、API Key 为空或返回字段为空时，使用本地兜底文案，并设置 `aiGenerated=false`。

### 5.5 安全约束

- DeepSeek API Key 只允许通过后端环境变量 `DEEPSEEK_API_KEY` 配置。
- 前端不得出现 DeepSeek Key。
- Git 不提交真实 Key。
- DeepSeek 调用失败不得影响推荐功能。

## 6. 前端设计

### 6.1 智能推荐页展示结构

推荐成功后右侧结果区展示：

1. AI 状态标签：
   - `AI 已生成分析`
   - 或 `规则推荐分析`
2. `aiSummary`：本次推荐整体解释。
3. 三个统计卡片：命中菜谱、累计热量、平均蛋白质。
4. `aiHealthTip`：健康建议。
5. `aiShoppingTip`：购物清单建议。
6. 生成购物清单按钮。
7. 推荐菜谱卡片列表。

### 6.2 菜谱图片策略

所有推荐卡片图片来自 `recipe.imageUrl`，即数据库正式菜谱绑定图片。

图片路径优先级：

1. 数据库 `recipe.image_url`。
2. 前端本地图片映射。
3. 默认占位图。

AI 不参与图片选择。

### 6.3 用户感知

当 `aiGenerated=true` 时，前端文案强调“AI 已结合你的输入生成分析”。

当 `aiGenerated=false` 时，前端文案显示“当前使用规则推荐分析”，避免用户误以为已经调用 AI。

## 7. 验收标准

### 7.1 功能验收

- 未配置 `DEEPSEEK_API_KEY` 时，推荐接口仍能返回菜谱和本地兜底分析。
- 配置 `DEEPSEEK_API_KEY` 时，推荐接口能返回 `aiGenerated=true`。
- 推荐菜谱必须来自数据库。
- 菜谱图片必须来自数据库图片路径或本地占位图。
- 生成购物清单时必须排除用户已有食材。
- 推荐历史能保存并展示 AI 分析字段。

### 7.2 演示验收

推荐页应能讲清楚：

- “这不是 AI 随机编菜，而是知识库增强推荐。”
- “数据库负责可信菜谱和营养，AI 负责解释推荐结果。”
- “AI 失败时系统仍可用。”
- “图片与菜谱一一绑定，避免图文不一致。”

## 8. 答辩表述

可在答辩中使用以下表述：

> 膳哉采用知识库增强的推荐 Agent 架构。菜谱、食材、营养和图片来自结构化数据库，系统先用规则算法完成过滤和评分，再将推荐结果交给 DeepSeek 生成个性化解释。这样既能体现 AI 能力，又避免 AI 编造菜谱、营养数据和图片不匹配的问题。

## 9. 当前决策

- 当前版本做知识库增强推荐 Agent。
- 当前版本不做 AI 随机生成正式菜谱。
- DeepSeek 只生成推荐分析，不生成数据库主数据。
- 图片只绑定正式菜谱，不绑定 AI 随机文本。
- 购物清单只基于数据库食材关系计算。
