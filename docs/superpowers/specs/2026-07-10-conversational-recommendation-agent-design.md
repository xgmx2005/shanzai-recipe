# 膳哉对话式智能推荐 Agent 设计

## 1. 背景

当前智能推荐页将条件表单与推荐结果同时放在同一屏幕，输入阶段占用了结果展示空间，也弱化了“智能助手”的产品感。当前后端已经具备规则推荐、健康档案合并、DeepSeek 推荐说明、推荐历史、菜谱详情和购物清单能力，但前端仍直接组装 `RecommendationRequest`，没有对话状态、自然语言纠错和刷新恢复能力。

本设计将推荐流程改为两个按顺序出现的独立页面：

1. 对话输入页：Agent 逐步理解并确认用户需求。
2. 推荐结果页：展示知识库真实菜谱、匹配解释和后续操作。

系统不生成新菜谱，不为动态文本强行匹配不准确的成品图。DeepSeek 负责理解、追问与解释，后端规则负责状态、安全、数据校验、知识库检索和数量计算。推荐完成时，系统可以通过受控的联网证据工具检索权威资料，由 DeepSeek 基于检索结果生成带出处的个性化讲解；菜谱详情继续使用稳定、可审核的知识库内容。

## 2. 目标

- 让用户用自然语言表达想吃什么、已有食材、忌口和用餐条件。
- 将自由表达稳定转换为现有 `RecommendationRequest` 所需的结构化字段。
- 对乱码、无关回答、模糊食材、字段冲突和 AI 异常作出明确反应。
- 只从现有菜谱知识库检索真实菜谱，并使用菜谱绑定的真实图片。
- 将对话输入与推荐结果完全分离，提升结果页的信息承载能力。
- 支持会话刷新恢复、重复请求幂等和推荐结果刷新恢复。
- 按目标人数缩放食材、总营养和购物清单数量。
- 在推荐结果中生成“为什么推荐、这道菜有什么好处”的个性化联网讲解，并保存可追溯来源。
- 提高详情页信息密度，为知识库菜谱提供稳定的营养好处、备菜说明和详细步骤。

## 3. 非目标

- 不生成知识库外的新菜谱。
- 不生成菜谱成品图。
- 不在本期建设完整的家庭库存管理系统。
- 不要求用户为每一种已有食材录入精确克数。
- 不让大模型自行放宽过敏、忌口或其他安全限制。
- 不用复杂动画掩盖等待时间，优先保证响应速度和状态清晰。
- 本期不实现“开始烹饪”专注模式、步骤倒计时、屏幕常亮或逐步导航。

## 4. 用户体验

### 4.1 对话输入页

路由为 `/user/recommend`。页面保持单列居中布局，不同时展示推荐结果。

Agent 按以下顺序补全信息：

1. `INTENT`：饮食意图，例如清淡、控热量、增肌或想吃的类别。
2. `INGREDIENTS`：现有主要食材，以及用户主动提供的数量和单位。
3. `RESTRICTIONS`：本次忌口、过敏和明确排除食材。
4. `CONTEXT`：人数和最大烹饪时间。
5. `CONFIRM`：展示可编辑的本次条件确认卡。

用户可以在一条消息中回答多个阶段的信息。系统提取所有可确认字段，跳过已经完成的阶段，只追问仍缺少或存在歧义的内容。

确认卡包含：

- 饮食目标与口味意图。
- 规范化后的已有食材及已知库存数量。
- 忌口、过敏和字段来源。
- 用餐人数与烹饪时间。
- “修改条件”和“确认并开始匹配”两个操作。

用户确认前不会创建推荐历史。确认成功后跳转到独立结果页。

### 4.2 推荐结果页

路由为 `/user/recommend/result?historyId={id}`，由独立的 `RecommendationResultView.vue` 承载，不再复用推荐历史列表组件。

页面展示：

- 本次确认条件与推荐编号。
- AI 生成的匹配说明、健康提示和购物提示。
- 基于受控联网检索生成的“为什么适合你”和“菜肴好处”，每条讲解必须绑定来源。
- 可展开的来源列表，展示标题、发布机构、链接、可获得的发布日期和本次访问时间。
- 推荐菜谱数量和排序依据。
- 每道菜谱的真实图片、匹配度、已利用食材、缺少食材、每人营养和用时。
- 查看详情、收藏和生成购物清单操作。
- 返回修改条件与重新开始推荐操作。

联网讲解只在本次推荐生成时执行一次，讲解与来源快照保存到推荐历史。结果页通过推荐历史详情接口恢复数据，刷新和查看历史时不会重复联网，也不会因网页内容变化而改变当时的解释。

### 4.3 菜谱详情页

详情页展示知识库中的稳定内容，不在每次打开时重新生成配料或做法：

- 缩短当前占用过多空间的首图区域，在首屏同时展示菜谱摘要、知识库好处、适用场景和关键操作。
- 营养区明确区分每人数据与本餐总计，并随目标人数联动。
- “为什么值得吃”展示经过维护的知识库好处，不冒充本次个性化医疗建议。
- “食材贡献”解释主要食材在本菜谱中的营养和口感作用。
- 详细步骤包含步骤标题、具体操作、预计时间、火候、完成判断和易错提示。
- 展示备菜说明、所需厨具、替换建议、购物清单和收藏操作。
- 不显示没有资源的图片占位框；没有步骤图时直接使用文字与食材图标。

## 5. 视觉规范

页面沿用现有膳哉设计系统，不创建新的色彩或组件语言：

- 内容宽度沿用 `1180px` 页面容器。
- 使用 `--sz-bg-soft`、`--sz-surface`、`--sz-evergreen`、`--sz-green-dark`、`--sz-mint` 和 `--sz-line`。
- 卡片圆角使用现有 `14px`，主要面板使用 `22px`。
- 使用薄米色边框和克制的 `--sz-shadow-soft`。
- 保留现有顶部导航、品牌区、图标体系与响应式断点。
- 对话气泡只区分 Agent 与用户，不使用大面积渐变和重动画。
- 桌面和移动端均保持单列对话；确认卡在移动端变为单列字段和全宽按钮。
- 结果页使用完整内容区域展示推荐，不保留聊天栏或条件侧栏。
- 结果页的 AI 讲解使用全宽信息区，来源默认折叠，避免引用信息压过菜谱本身。
- 详情页压缩首屏留白，在同一视觉高度内呈现图片、摘要、知识库好处和主要操作。
- 食材与步骤区域保持清晰双栏关系，步骤列获得更多宽度；移动端按好处、食材、步骤顺序单列展示。

## 6. 架构边界

采用“后端状态机 + DeepSeek 结构化理解 + 现有推荐服务 + 受控联网证据工具”的架构。

### 6.1 DeepSeek 负责

- 从自然语言提取饮食意图、食材、数量、忌口、人数和时间。
- 识别模糊词、未知词和需要进一步确认的内容。
- 生成符合当前阶段的自然语言回复。
- 在推荐结果生成后解释匹配原因、健康提示和购物提示。
- 根据已推荐菜谱与用户目标生成有限数量的证据检索请求。
- 只基于后端返回的证据片段生成个性化好处，并为每条陈述绑定 `evidenceIds`。

### 6.2 后端规则负责

- 决定当前阶段和下一条问题。
- 校验会话所有权、消息幂等和状态迁移。
- 使用食材库、别名词典和单位规则规范化输入。
- 合并健康档案，并处理本次条件与档案的冲突。
- 将过敏和忌口作为不可放宽的硬约束。
- 判断会话是否达到确认条件。
- 组装现有 `RecommendationRequest` 并调用 `RecommendationService`。
- 从知识库检索、评分和保存推荐历史。
- 计算人数缩放、营养总量和购物清单数量。
- 执行联网检索、来源白名单过滤、超时控制、结果去重和证据快照保存。
- 校验 AI 讲解中的每个 `evidenceId` 确实来自本次检索结果，并拦截医疗化或无来源陈述。

大模型输出不直接写入推荐条件，必须通过后端校验后才能进入确认状态。

### 6.3 联网证据工具

DeepSeek API 通过 Tool Calls 请求 `search_health_evidence`，模型本身不直接访问网页。后端提供 `HealthEvidenceSearchClient` 接口，首个实现适配 Tavily Search API，API 密钥只通过后端环境变量 `TAVILY_API_KEY` 注入。

检索规则：

- 每次推荐最多生成 3 个检索问题，围绕已命中菜谱的主要营养事实和用户目标。
- 优先允许政府卫生机构、正式膳食指南、大学、权威医院、国际公共营养机构和同行评议论文来源。
- 拒绝电商、营销软文、匿名聚合站和无法确认发布机构的来源。
- 对结果做域名白名单、链接协议、标题、正文片段和重复 URL 校验。
- 搜索与内容获取设置总超时；失败时降级到知识库固定说明，不阻塞真实菜谱结果。
- 发送给 DeepSeek 的证据使用内部 ID，最终讲解只能引用这些 ID，不能自行创建链接。

AI 讲解采用结构化格式：

```json
{
  "overview": "本次优先选择高蛋白、热量适中的菜谱。",
  "recipes": [
    {
      "recipeId": 1,
      "personalizedReason": "符合你控制本餐热量并提高蛋白质摄入的目标。",
      "benefits": [
        {
          "text": "本菜谱每人约含 35g 蛋白质，可作为本餐的主要蛋白质来源。",
          "evidenceIds": ["source-1"]
        }
      ]
    }
  ]
}
```

没有有效证据的好处不得进入结果。系统不使用“治疗、治愈、降血糖、预防疾病”等医疗化结论替代专业建议。

## 7. 会话状态模型

### 7.1 `recommendation_conversation`

- `id BIGINT`：会话标识。
- `user_id BIGINT`：所属用户。
- `stage VARCHAR(30)`：`INTENT`、`INGREDIENTS`、`RESTRICTIONS`、`CONTEXT` 或 `CONFIRM`。
- `status VARCHAR(30)`：`ACTIVE`、`READY_TO_CONFIRM`、`COMPLETED` 或 `CANCELLED`。
- `context_json JSON`：已校验的结构化上下文。
- `invalid_answer_count INT`：当前阶段连续无效回答次数。
- `recommendation_history_id BIGINT NULL`：确认后关联的推荐历史。
- `created_at DATETIME`、`updated_at DATETIME`。

`context_json` 至少包含：

```json
{
  "intentText": "清淡并控制热量",
  "dietGoal": "FAT_LOSS",
  "availableIngredients": [
    {"name": "鸡胸肉", "quantity": 300, "unit": "g", "quantityKnown": true},
    {"name": "鸡蛋", "quantity": 2, "unit": "个", "quantityKnown": true},
    {"name": "西兰花", "quantityKnown": false}
  ],
  "excludedIngredients": ["香菜"],
  "allergyIngredients": ["牛奶"],
  "cookingTime": 30,
  "servings": 1,
  "sources": {
    "dietGoal": "CONVERSATION",
    "allergyIngredients": "PROFILE"
  }
}
```

### 7.2 `recommendation_conversation_message`

- `id BIGINT`：消息标识。
- `conversation_id BIGINT`：所属会话。
- `client_message_id VARCHAR(64) NULL`：客户端幂等标识。
- `role VARCHAR(20)`：`USER` 或 `ASSISTANT`。
- `content TEXT`：原始消息。
- `extracted_data_json JSON NULL`：本轮提取结果和置信度。
- `created_at DATETIME`。

同一会话内的 `client_message_id` 必须唯一。网络重试只能返回已保存的处理结果，不能重复推进阶段。

### 7.3 推荐历史联网讲解扩展

`recommendation_history` 增加：

- `ai_benefit_analysis JSON NULL`：本次个性化推荐讲解。
- `ai_evidence_sources JSON NULL`：本次使用的来源快照。
- `ai_evidence_status VARCHAR(20)`：`COMPLETED`、`FALLBACK` 或 `FAILED`。
- `ai_evidence_fallback_reason VARCHAR(255) NULL`：无法完成联网讲解时的降级原因。

来源快照包含内部证据 ID、标题、机构、URL、可为空的发布日期、访问时间和用于生成讲解的正文片段。历史页读取快照，不重新检索。

### 7.4 详情页知识库扩展

新增 `recipe_benefit`：

- `recipe_id`、`title`、`description`、`fact_basis`、`target_goals`、`sort_order`。

新增 `recipe_step`：

- `recipe_id`、`step_number`、`title`、`instruction`、`duration_minutes`、`heat_level`、`completion_cue`、`tips`。

现有 45 道菜谱需要迁移为结构化步骤。每道菜保持与实际复杂度相符的 4 至 8 个步骤，不使用“煮熟即可”之类无法判断完成状态的笼统描述。

## 8. 接口设计

### 8.1 创建会话

`POST /api/recommendation-conversations`

创建会话、读取健康档案并返回第一条 Agent 提问。若用户存在未完成会话，前端可显式选择继续或重新开始，不自动合并两个会话。

### 8.2 发送消息

`POST /api/recommendation-conversations/{id}/messages`

请求包含 `clientMessageId` 和 `content`。响应包含：

- 用户消息和 Agent 回复。
- 当前 `stage` 与 `status`。
- 已确认的结构化上下文。
- `missingFields`、`unknownTerms` 和 `conflicts`。
- 是否应展示条件确认卡。
- AI 是否成功参与本轮理解。

### 8.3 恢复会话

`GET /api/recommendation-conversations/{id}`

返回会话、消息和结构化上下文。仅会话所属用户可读取。

### 8.4 修改确认条件

`PATCH /api/recommendation-conversations/{id}/context`

修改确认卡中的单个或多个字段。修改后重新执行规范化、冲突和完整性校验，不能绕过安全规则。

### 8.5 确认并推荐

`POST /api/recommendation-conversations/{id}/confirm`

要求状态为 `READY_TO_CONFIRM`。后端再次验证所有硬约束，组装现有 `RecommendationRequest`，调用 `RecommendationService.recommend`，再执行受控联网证据检索与 DeepSeek 个性化讲解，保存 `recommendation_history_id`、讲解和来源快照，并返回扩展后的 `RecommendationResponse`。

推荐检索成功但联网证据阶段失败时，确认接口仍返回真实菜谱，并将 `aiEvidenceStatus` 标记为 `FALLBACK`。前端展示知识库固定说明和“本次未获取外部参考资料”，不能伪造来源。

重复确认已完成会话时，返回同一个推荐历史结果，不能创建重复历史。

## 9. 输入理解与差错控制

### 9.1 有效回答

- 一条消息可填充多个字段。
- 已确认字段不会因后续无关文本被静默覆盖。
- 同义词通过别名词典归一，例如“西蓝花”映射为“西兰花”。
- 模糊词如“肉”“青菜”不直接入库，必须追问具体类型。
- 未知词保留在 `unknownTerms` 中，用户确认或修正前不进入推荐条件。

### 9.2 乱码、乱打和答非所问

- 空白、纯符号、明显乱码或与当前任务无关的回答不推进状态，也不写入结构化上下文。
- 第一次无效回答：Agent 说明未识别到与当前问题相关的信息，并提供一个回答示例。
- 同一阶段第二次连续无效：显示快捷选项或更明确的格式提示。
- 同一阶段第三次连续无效：提供重新开始或继续补充的操作，但不自动结束会话。
- 恢复有效回答后将当前阶段的连续无效计数清零。
- 冒犯性或提示词注入内容不回显、不执行，只返回任务范围内的中性提示。

### 9.3 冲突

- 本次对话与健康档案冲突时必须展示冲突并要求确认。
- 同一食材同时出现在已有食材与忌口中时，忌口优先，并阻止确认。
- 过敏是最高优先级硬约束，不能被普通偏好或推荐数量覆盖。
- 人数小于 1、烹饪时间无效或数量单位无法换算时要求重新确认。

### 9.4 AI 与网络异常

- DeepSeek 返回非法 JSON 时重试一次。
- 第二次仍失败、超时或未配置密钥时，切换到相同聊天界面的固定问题模式。
- 降级模式使用本地食材词典、别名映射、数字与单位解析；无法可靠解析时使用快捷选项确认，不猜测数据。
- 联网证据搜索、来源读取或讲解生成失败时，只降级推荐讲解，不撤销已经通过安全过滤的真实菜谱结果。
- 搜索结果全部被白名单过滤掉时，不让 DeepSeek依据模型记忆补充“出处”。
- 前端使用同一 `clientMessageId` 重试超时消息，后端保证幂等。
- 会话刷新后从后端恢复，不依赖 Vue 组件内存。

## 10. 安全推荐原则

系统原则是“安全约束优先于推荐数量”。

- 推荐只检索状态有效的知识库菜谱。
- 完全匹配优先；不存在完全匹配时可以展示缺少少量食材的现有菜谱。
- 推荐结果必须明确展示已利用食材和仍需购买食材。
- 如果没有通过过敏和忌口过滤的菜谱，返回“暂无安全推荐”并引导修改条件。
- 无结果时不得生成新菜谱，也不得自动删除硬约束。
- 菜谱图片来自该知识库菜谱绑定的真实图片，不按用户自由文本猜图。
- 联网资料只用于解释已推荐菜谱，不参与绕过过敏过滤，也不能修改知识库中的配料和步骤。

## 11. 人数与数量缩放

现有代码只将 `servings` 保存到推荐历史，菜谱详情与购物清单仍使用数据库原始数量。本功能需要补齐真正的份量缩放。

每道菜谱的 `recipe.servings` 是基础人数：

```text
缩放系数 = 目标人数 / 菜谱基础人数
目标食材量 = 基础食材量 × 缩放系数
```

规则如下：

- 克、毫升等连续单位按比例计算并使用合理精度。
- 鸡蛋、馒头等离散单位向上取整。
- 盐、油和“适量”类调料显示“约”或保留“适量”，避免虚假精确。
- 菜谱卡与详情页区分“每人营养”和“本餐总计”。
- 推荐评分使用每人营养，避免人数增加导致同一菜谱被错误判定为高热量。
- 购物清单先按人数缩放，再合并多道菜谱的同类食材。
- 用户明确提供库存数量时，从目标数量中扣除；只提供食材名称时标记“库存数量未知”，提示用户核对，不直接认定完全足够。
- Agent 不强制追问所有食材克数，只在数量会影响菜谱可执行性或购物清单准确性时追问关键食材。

## 12. 前端组件边界

- `RecommendView.vue`：对话容器、会话创建和恢复、消息发送、确认卡展示。
- `RecommendationMessageList.vue`：纯消息展示，不直接调用接口。
- `RecommendationComposer.vue`：输入、发送、快捷选项和等待状态。
- `RecommendationConditionSummary.vue`：确认卡及字段修改入口。
- `RecommendationResultView.vue`：独立结果页，通过 `historyId` 加载数据。
- `RecipeRecommendationCard.vue`：展示真实图片、匹配解释、已利用和缺少食材。
- `RecommendationEvidencePanel.vue`：展示个性化讲解、证据状态和可展开来源。
- `RecipeBenefitSection.vue`：展示详情页稳定知识库好处与事实依据。
- `RecipeStepTimeline.vue`：展示结构化详细步骤、火候、完成判断和易错提示。
- `servingScale` 工具或后端领域服务：统一份量、营养与购物数量换算，避免前后端各自计算出不同结果。

前端只展示后端确认后的结构化上下文，不自行决定阶段、安全冲突或最终食材名称。

## 13. 推荐与购物闭环

```text
创建对话
→ 多轮回答与校验
→ 条件确认
→ RecommendationService 检索真实菜谱
→ 受控联网检索权威资料
→ DeepSeek 生成带证据 ID 的个性化讲解
→ 保存 recommendation_history 与来源快照
→ 跳转独立结果页
→ 查看菜谱详情
→ 按人数生成购物清单
→ 跳转新购物清单
```

详情页从推荐结果进入时携带 `historyId` 或目标人数，使份量与推荐上下文保持一致。购物清单创建请求需要增加目标人数或明确的缩放后需求，避免继续使用数据库基础份量。

## 14. 隐私与权限

- 所有会话接口要求登录。
- 用户只能访问自己的会话、消息和推荐历史。
- 原始对话仅用于本项目的推荐流程，不发送到前端以外的第三方服务，DeepSeek 调用除外；证据搜索只发送去标识化的菜谱和营养关键词，不发送用户姓名或完整健康档案。
- DeepSeek API 密钥继续只存在于后端环境变量。
- Tavily 搜索密钥继续只存在于后端环境变量。
- 日志不记录 API 密钥、完整健康档案或完整原始对话正文。
- 对话内容和提取结果需要长度限制，防止超大输入和数据库滥用。

## 15. 测试与验收

### 15.1 后端单元测试

- 状态机按顺序推进五个阶段。
- 一次回答多个字段时跳过已完成阶段。
- 空内容、乱码、乱打和无关回答不推进状态。
- 连续无效回答触发示例、快捷选项和重新开始提示。
- 同义词归一、模糊词追问和未知词隔离。
- 健康档案合并、忌口冲突和过敏硬拦截。
- 重复 `clientMessageId` 和重复确认保持幂等。
- DeepSeek 成功、非法 JSON、超时和禁用时的降级路径。
- 联网证据检索成功、超时、无白名单来源和来源去重。
- AI 讲解只能引用本次证据 ID，无证据陈述被拒绝。
- 推荐历史保存并恢复讲解与来源快照，不重复联网。
- 无完全匹配、无安全推荐和不生成新菜谱。
- 1、2、4 人份的连续单位、离散单位和营养换算。
- 购物清单缩放、合并与已知库存扣减。
- 结构化菜谱好处和详细步骤按顺序返回。

### 15.2 控制器与持久化测试

- 未登录用户不能访问会话接口。
- 用户不能读取或修改他人的会话。
- 刷新后恢复消息、阶段和结构化上下文。
- 完成会话正确关联推荐历史。
- 数据库迁移可以从现有 `schema.sql` 和当前迁移状态执行。

### 15.3 前端测试

- 创建会话并渲染第一条提问。
- 发送消息、等待状态、失败重试和快捷选项。
- 条件确认卡的修改和确认。
- 确认后跳转 `/user/recommend/result?historyId={id}`。
- 结果页刷新后通过历史接口恢复数据。
- 推荐页展示动态联网讲解、降级状态和可展开来源。
- 返回修改条件和重新开始推荐。
- 人数相关的每人营养、总营养和食材数量展示。
- 详情页首屏不再出现大面积无信息留白，并展示知识库好处、食材贡献和详细步骤。
- 桌面与移动端不出现横向溢出、遮挡和双栏输入结果混排。

### 15.4 端到端验收

至少验证以下路径：

1. 正常自然语言对话到真实菜谱结果。
2. 一条消息包含多个条件。
3. 连续乱输入后恢复正常回答。
4. 模糊食材和健康档案冲突。
5. 无安全推荐后返回修改条件。
6. DeepSeek 成功调用与关闭密钥后的降级模式。
7. 联网讲解包含有效来源，搜索不可用时显示知识库降级说明。
8. 推荐结果到高信息密度详情，再按人数生成购物清单。

## 16. 成功标准

- 输入页与结果页是两个按顺序出现的独立页面。
- 用户无需操作传统多字段表单即可完成推荐条件收集。
- 任何无效或模糊输入都有可理解的提示，不会静默污染条件。
- 推荐结果全部来自知识库，并使用对应真实菜谱图片。
- 推荐页的动态菜肴好处都有本次保存的有效来源，刷新和历史查看时内容保持一致。
- 详情页展示稳定知识库好处和可执行的详细步骤，不依赖实时 AI 重新编写。
- 过敏和忌口冲突不会出现在推荐结果中。
- 会话和结果刷新后均可恢复。
- 人数变化会同步改变食材、总营养和购物清单数量，且推荐菜谱数量不因人数增加。
- DeepSeek 不可用时仍可完成受控推荐流程。
