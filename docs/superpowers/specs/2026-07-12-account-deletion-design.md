# 账号注销设计规格

## 背景

膳哉后续可能上线给真实朋友使用，账号注销不能停留在前端清登录态。注销需要同时满足用户侧“不能再登录”、数据侧“不破坏外键关系”、隐私侧“移除个人识别信息”和交互侧“避免误操作”。

## 目标

- 普通用户可以在健康档案页主动注销自己的账号。
- 注销后当前登录态立即失效，前端跳转登录页。
- 注销账号不能再次登录，也不能继续通过旧 token 访问 `/api/auth/me`。
- 注销会释放原用户名，后续可重新注册相同用户名。
- 注销会清除头像文件、头像 URL、昵称和健康档案敏感信息。
- 管理员账号不能通过用户端注销，避免误删维护账号。

## 非目标

- 不实现 30 天冷静期和账号恢复流程。
- 不实现邮件验证或短信确认。
- 不物理删除 `user` 记录，避免破坏推荐历史、购物清单、菜谱创建人等外键关系。
- 不删除已完成的推荐历史和购物清单历史，只做账号匿名化。

## 后端设计

### 接口

新增 `DELETE /api/auth/me`。

请求需要登录。后端从 `JwtUser` 取当前用户 ID，不接受前端传用户 ID。

成功返回：

```json
{
  "code": 200,
  "message": "success",
  "data": null
}
```

### 注销规则

注销时读取当前用户：

- 用户不存在或 `status != 1`：抛出业务异常。
- `role = MAINTAINER`：拒绝注销。
- `role = USER`：允许注销。

注销写入用户表：

- `status = 0`
- `username = deleted_{userId}_{timestamp}`
- `nickname = 已注销用户`
- `avatar_url = ''`
- `avatar_theme = leaf`

注销清理：

- 调用头像存储服务删除原头像文件。
- 清空健康档案敏感字段：性别、年龄、身高、体重、BMI、饮食目标、口味偏好、忌口、过敏、烹饪时间偏好、目标热量、档案完成状态。
- 删除收藏关系。

### 登录与当前用户保护

- `login` 必须拒绝 `status != 1` 的用户。
- `currentUser` 和账号资料更新接口必须拒绝 `status != 1` 的用户，避免旧 JWT 继续访问。

## 前端设计

### 入口位置

在健康档案页“账号资料”下方增加“危险操作”区域。视觉保持克制，不抢健康档案主体。

### 确认交互

点击“注销账号”后打开确认弹窗：

- 明确说明注销后不能继续登录，健康档案和头像会被清理。
- 要求用户输入 `注销账号` 四个字后才能确认。
- 确认按钮在输入正确前禁用。
- 成功后调用 `auth.logout()` 并跳转 `/login`。

### 失败处理

- 接口失败时保留当前页面并显示错误提示。
- 管理员或异常账号触发失败时展示后端返回消息。

## 测试设计

### 后端

- 注销普通用户会禁用账号、释放用户名、匿名昵称、清空头像。
- 注销会调用头像删除，并清空健康档案、删除收藏。
- 管理员账号注销会被拒绝。
- 已注销账号不能登录。
- 已注销账号不能读取当前用户信息。

### 前端

- 健康档案页包含危险操作区和注销入口。
- 确认弹窗要求输入 `注销账号`。
- Store 提供 `deleteAccount`，成功后清空登录态。
- API 使用 `DELETE /auth/me`。

## 风险与取舍

该方案不是完全合规级账号删除，但已经满足小规模上线的基础要求：用户不能继续使用账号，个人识别信息被清理，业务历史保持匿名可追溯。后续如果正式公开上线，可以再增加冷静期、邮件确认、审计日志和后台账号恢复能力。
