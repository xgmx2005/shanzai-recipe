# 膳哉首页工作台设计规格

## 背景

膳哉当前已经具备健康档案、对话式智能推荐、推荐结果、菜谱详情、收藏菜谱和购物清单等主要功能。首页不应该再承担复杂操作，也不应该像普通营销页一样只做品牌展示。首页的核心价值是让用户一进入系统就知道今天该怎么开始，并能快速进入主要功能闭环。

本次首页改版采用「今日饮食工作台」方向：用更克制、更有秩序的界面组织健康状态、推荐入口、使用流程和可复用菜谱。

## 设计目标

1. 首页第一屏明确表达膳哉的产品能力：结合健康档案、饮食目标、时间、忌口和用户表达，生成可执行的菜谱推荐。
2. 首屏视觉重心指向「开始智能推荐」，减少重复入口和无效装饰。
3. 将健康档案状态作为推荐前置条件展示，但不喧宾夺主。
4. 通过流程区展示项目完整闭环：表达需求、生成推荐、整理购物清单。
5. 首页菜谱卡作为灵感展示和知识库入口，每次进入页面随机展示，避免固定模板感。
6. 保持现有项目的现代轻健康视觉风格：暖白底、深绿主色、薄荷绿辅助、少量谷物黄/番茄红点缀。

## 信息架构

首页分为四个主要区域。

### 1. 今日饮食工作台首屏

首屏采用左右结构。

左侧为主行动区：

- 标签：`膳哉智能轻饮食助手`
- 标题：`今天这顿，交给膳哉来配`
- 说明：强调系统会结合健康档案、饮食目标、烹饪时间、忌口和用户表达，生成清楚可执行的推荐。
- 主按钮：`开始智能推荐`，跳转 `/user/recommend`
- 次按钮：`完善健康档案`，跳转 `/user/profile`

右侧为今日状态面板：

- BMI 与状态
- 今日目标热量
- 饮食目标
- 推荐记录数量或最近推荐入口

状态面板不展示大段说明，重点是让用户确认系统“知道我的身体和目标”。

### 2. 使用路径区

将原有「快速入口」改成三步流程，不再像普通功能卡片：

1. `说出想吃什么`：进入智能推荐，用户通过对话表达已有食材、口味、时间和人数。
2. `生成推荐菜谱`：系统结合知识库与健康档案给出推荐结果和理由。
3. `整理购物清单`：用户可从推荐结果、详情页或收藏页生成购物清单。

每一步使用简洁图标、短标题和一句说明，视觉上形成从左到右的流程。

### 3. 首页灵感菜谱区

展示 4 张知识库菜谱卡，标题建议为 `今天也可以这样吃`。

菜谱卡内容：

- 菜谱图片
- 菜谱名称
- 热量
- 烹饪时间
- 收藏按钮
- 查看详情入口

展示规则：

- 每次进入首页时，从 `listRecipes()` 返回的知识库菜谱中随机抽取最多 4 道。
- 随机仅用于首页灵感展示，不影响 AI 推荐排序、推荐历史或购物清单生成。
- 如果菜谱数量少于 4 道，则展示全部。
- 收藏状态仍然按当前账号读取，随机展示不能破坏收藏/取消收藏逻辑。

### 4. 辅助状态

- 加载中：使用骨架屏，不出现大面积空白。
- 部分接口失败：用轻量提示说明部分数据暂不可用；只要菜谱或基础档案仍可展示，首页不整体崩溃。
- 无菜谱：显示空状态，引导用户去智能推荐或等待管理员维护菜谱库。

## 视觉原则

首页应比当前版本更像“工作台”，而不是“卡片集合”。

- 首屏高度控制在常见桌面首屏内，避免用户刚进入就需要滚动才能理解重点。
- 主按钮必须是全页最明显的行动按钮。
- 右侧状态面板使用小卡片组合，信息密度高但不拥挤。
- 流程区使用横向步骤，不使用重复的大 Banner。
- 菜谱区视觉弱于首屏，但图片质量要足够好，卡片留白克制。
- 圆角保持 14px 到 20px 区间，不使用过大的胶囊式容器。
- 色彩继续沿用深绿、薄荷绿、暖白、谷物黄，红色只用于收藏等情感操作。

## 交互规则

- 点击 `开始智能推荐` 进入对话式推荐页。
- 点击 `完善健康档案` 进入健康档案页。
- 点击流程区第一步进入智能推荐页。
- 点击流程区第三步进入购物清单页。
- 点击菜谱卡进入菜谱详情页。
- 点击菜谱卡收藏按钮时，不触发卡片详情跳转。
- 已收藏菜谱显示实心红心，未收藏显示轻量心形。
- 收藏或取消收藏后，应刷新当前账号的收藏状态，但不需要重新随机菜谱列表。

## 数据与实现边界

首页继续复用现有接口：

- `auth.loadProfile()`
- `getProfileSummary()`
- `listRecipes()`
- `listRecommendationHistory()`
- `listFavorites()`
- `favoriteRecipe()`
- `unfavoriteRecipe()`

随机菜谱卡在前端完成即可，不新增后端接口。实现时在首页加载菜谱后做一次随机抽样，保存在 `recipes` 状态中。不要在模板渲染时直接随机，避免响应式更新导致卡片跳动。

推荐次数文案需要避免误导。如果仍然使用 `listRecommendationHistory()` 的总数量，则文案应写成 `累计推荐记录`，不要写 `最近 7 天生成推荐`。

## 测试要求

新增或更新首页测试，至少覆盖：

1. 首页采用 `home-workbench`、`daily-status-panel`、`workbench-flow`、`inspiration-recipes` 等结构标识。
2. 主行动按钮跳转智能推荐页，次行动按钮跳转健康档案页。
3. 首页菜谱列表经过随机抽样函数处理，不是固定 `slice(0, 4)`。
4. 随机抽样函数最多返回 4 道菜，菜谱不足时返回全部。
5. 推荐次数文案不再出现 `最近 7 天生成推荐`。

## 不做范围

- 不改智能推荐核心逻辑。
- 不改菜谱知识库排序规则。
- 不新增后端随机菜谱接口。
- 不重做用户导航栏。
- 不在首页加入复杂动画或全屏转场。

## 验收标准

1. 用户打开首页后，3 秒内能看出主功能是智能菜谱推荐。
2. 首页第一屏能同时看到主按钮和健康状态面板。
3. 菜谱灵感区每次进入页面有机会出现不同菜谱组合。
4. 首页视觉与健康档案、智能推荐、详情页保持同一套轻健康风格。
5. 前端单元测试和生产构建通过。
