# 注册后健康档案引导设计

## 背景

当前用户注册成功后直接进入 `/user/home`。健康档案页已经具备完整编辑能力，但它更像设置中心，不适合作为新用户第一次进入产品时的引导。膳哉的核心卖点是“结合身体数据、饮食目标和忌口生成推荐”，因此新用户注册后需要先被轻量引导配置健康档案。

## 目标

- 注册成功后，引导普通用户完成一份基础健康档案，再进入首页。
- 让用户理解健康档案会影响推荐质量，而不是把它当成可有可无的表单。
- 保留“稍后完善”入口，避免用户被强制流程卡住。
- 不改变管理员注册或登录后的跳转逻辑。
- 不重做现有健康档案设置页，只新增首次引导流程并复用既有字段和保存接口。

## 非目标

- 不新增复杂医学评估、疾病问卷或营养诊断。
- 不要求上传头像或编辑账号资料。
- 不把健康档案引导做成聊天式 Agent，避免注册链路过长。
- 不为每个字段增加后端新表。优先复用现有 profile 保存能力，只在需要时增加完成状态字段。

## 用户流程

1. 用户在 `/register` 注册成功。
2. 若注册结果为普通用户，跳转到 `/user/onboarding`。
3. 用户看到三步式健康档案引导：
   - 第一步：基础身体信息。
   - 第二步：饮食目标与烹饪时间。
   - 第三步：口味偏好、忌口和过敏食材。
4. 用户点击“保存并开始使用”后，系统保存健康档案并标记引导完成，然后跳转 `/user/home`。
5. 用户点击“稍后完善”后，系统标记本次会话已跳过并跳转 `/user/home`，但不标记档案完成。
6. 首页和智能推荐页在档案未完成时展示轻量提示，引导用户回到 `/user/onboarding` 或 `/user/profile` 完善。

## 页面设计

页面名称：健康档案引导页  
路由：`/user/onboarding`  
建议组件：`ProfileOnboardingView.vue`

整体视觉沿用膳哉的现代轻健康风格：

- 背景使用浅米白与浅绿色渐变，不使用沉重全屏弹窗。
- 主体为居中宽卡片，左侧是步骤说明，右侧是当前步骤表单。
- 顶部显示 `1 / 3`、`2 / 3`、`3 / 3` 的进度，降低用户心理负担。
- 每一步都用一句短文案说明字段价值，例如“这些信息只用于估算推荐热量范围”。
- 主按钮使用绿色：“下一步”“保存并开始使用”。
- 次按钮使用文本按钮：“稍后完善”。

## 三步字段

### 第一步：基础身体信息

字段：

- 身高 `heightCm`
- 体重 `weightKg`
- 年龄 `age`
- 性别 `gender`

校验：

- 使用现有健康档案页的数值范围。
- 必填，缺失时给出具体提示。

说明文案：

> 用于估算 BMI 和每日热量范围，帮助推荐更贴合你的身体状态。

### 第二步：饮食目标

字段：

- 饮食目标 `dietGoal`
- 烹饪时间偏好 `cookingTimePreference`

目标选项：

- 日常健康
- 减脂控热量
- 健身增肌

说明文案：

> 饮食目标会影响推荐排序，烹饪时间会过滤不适合当前时间的菜谱。

### 第三步：口味与限制

字段：

- 口味偏好 `tastePreferences`
- 忌口食材 `avoidIngredients`
- 过敏食材 `allergyIngredients`

交互：

- 复用现有 `IngredientTagInput` 标签输入。
- 提供少量快捷标签，例如清淡、高蛋白、少油、不要辣、鸡蛋、牛肉。
- 过敏食材区域文案要更严肃，强调会参与安全过滤。

说明文案：

> 忌口和过敏会优先过滤，口味偏好会帮助 AI 更准确解释推荐理由。

## 完成状态

推荐增加用户档案完成状态：

- 前端状态名：`profileCompleted`
- 后端字段名建议：`profile_completed`
- 默认值：`false`
- 用户在 onboarding 点击“保存并开始使用”后置为 `true`
- 用户在完整健康档案页保存时，如果必填字段齐全，也可以置为 `true`

如果短期不想改后端结构，可以先用前端本地状态兜底：

- `localStorage` 记录 `profile_onboarding_skipped:<userId>`
- 根据当前 profile 字段完整度计算 `isProfileReady`

但正式方案建议后端落字段，因为换浏览器或重新登录后仍应知道用户是否完成过引导。

## 跳转规则

注册成功：

- 普通用户：`/user/onboarding`
- 管理员：`/admin/dashboard`

登录成功：

- 若普通用户且 `profileCompleted = false`，进入 `/user/home`，但首页展示轻量提示，不强制跳 onboarding。
- 若普通用户且 `profileCompleted = true`，进入 `/user/home`。
- 管理员维持 `/admin/dashboard`。

这样设计避免老用户每次登录被打断，也避免新用户注册后直接进入空首页。

## 首页和推荐页提示

当 `profileCompleted = false` 时：

- 首页顶部展示一个小提示条：“完善健康档案后，推荐会更准确。”
- 智能推荐页在开始对话前展示小提示：“当前使用默认档案，完善后会结合热量、忌口和口味偏好。”
- 提示条提供按钮：“去完善”。
- 用户关闭提示后，本次会话不再反复弹出。

## 数据与接口

复用现有健康档案保存接口：

- `getProfile`
- `saveProfile`

如增加完成状态，建议扩展：

- `Profile` 响应增加 `profileCompleted: boolean`
- `ProfileRequest` 可增加 `profileCompleted?: boolean`
- 后端保存档案时允许更新该字段

前端注册流程需调整：

- `RegisterView.vue` 注册成功后跳 `/user/onboarding`
- `ProfileOnboardingView.vue` 保存成功后跳 `/user/home`

## 错误处理

- 网络错误：保留用户已输入数据，不清空表单。
- 保存失败：页面顶部显示错误提示，并允许重试。
- 跳过：无需请求保存接口，直接进入首页。
- 必填字段不完整：停留当前步骤并提示具体字段。

## 测试点

- 注册成功后普通用户跳转 `/user/onboarding`。
- onboarding 三步标题和关键字段存在。
- 必填字段缺失时不能进入下一步。
- 点击“稍后完善”跳转 `/user/home`，不标记 `profileCompleted`。
- 点击“保存并开始使用”调用保存接口并跳转 `/user/home`。
- 首页在档案未完成时展示完善提示。
- 智能推荐页在档案未完成时展示默认档案提示。

## 实施顺序

1. 增加完成状态的数据模型和后端保存支持。
2. 新增 `/user/onboarding` 路由和页面。
3. 修改注册成功跳转。
4. 首页和智能推荐页增加轻量提示。
5. 补齐单元测试和构建验证。

## 自审结果

- 没有未定事项或占位符。
- 范围聚焦在注册后健康档案引导，不重做健康档案设置页。
- 跳过逻辑和完成状态明确，避免用户被强制卡住。
- 数据方案提供正式后端字段和短期前端兜底，但实施推荐以后端字段为准。
